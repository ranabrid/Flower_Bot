## UC_01: Установить регулярные напоминания

#### Цель
Установить регулярные напоминания о покупке цветов по удобному для пользователя расписанию

#### Инициатор
Пользователь

#### Предусловие:
- бот запущен на сервере;
- пользователь имеет активный аккаунт в Telegram

#### Триггер
Пользователь отправляет команду `/start`

#### Основной сценарий:
1. система отправляет приветственное сообщение и предложение начать настройку или отложить её;
2. пользователь соглашается;
3. система спрашивает, когда пользователь хочет получить напоминание;
4. пользователь вводит дату в любом формате (например, «Завтра», «5 мая 2026», «5.05.26», «Через час»);
5. система планирует первое напоминание на заданную дату;
6. система подтверждает успешное планирование.
7. система спрашивает, как часто пользователь хочет получать напоминания (с вариантами раз в одну/две/три недели или пользовательский вариант);
8. пользователь выбирает «Свой вариант»;
9. система просит указать периодичность в днях;
10. пользователь указывает желаемую частоту напоминаний;
11. система сохраняет периодичность в БД;
12. система отправляет подтверждение.

#### Альтернативный сценарий:

##### &nbsp; &nbsp; 2а Пользователь передумал
 &nbsp; &nbsp; &nbsp; &nbsp; 2а.1. пользователь нажимает «В другой раз»;  
 &nbsp; &nbsp; &nbsp; &nbsp; 2а.2 Система отправляет сообщение: «Хорошо, напишите `/start`, когда передумаете»;  
 &nbsp; &nbsp; &nbsp; &nbsp; 2а.3 Система завершает диалог.

##### &nbsp; &nbsp; 4а Некорректная дата
 &nbsp; &nbsp; &nbsp; &nbsp; 4а.1. пользователь ввел некорректные данные (любое слово, не связанное с датой или временем);  
 &nbsp; &nbsp; &nbsp; &nbsp; 4а.2. система сообщает об ошибке;  
 &nbsp; &nbsp; &nbsp; &nbsp; 4a.3. происходит возврат на шаг 4 основного сценария.

##### &nbsp; &nbsp; 4b Дата в прошлом
 &nbsp; &nbsp; &nbsp; &nbsp; 4b.1. пользователь ввел уже прошедшую дату;  
 &nbsp; &nbsp; &nbsp; &nbsp; 4b.2. система сообщает об ошибке;  
 &nbsp; &nbsp; &nbsp; &nbsp; 4b.3. происходит возврат на шаг 4 основного сценария.

##### &nbsp; &nbsp; 8а Выбор системного варианта
 &nbsp; &nbsp; &nbsp; &nbsp; 8а.1. пользователь выбирает один из существующих вариантов  
 &nbsp; &nbsp; &nbsp; &nbsp; 8а.2. переход к шагу 12 основного сценария.

##### &nbsp; &nbsp; 11а Данные введены в другом формате
 &nbsp; &nbsp; &nbsp; &nbsp; 11а.1. пользователь вводит не число;  
 &nbsp; &nbsp; &nbsp; &nbsp; 11а.2. система сообщает об ошибке;  
&nbsp; &nbsp; &nbsp; &nbsp; 11a.3. происходит возврат на шаг 10 основного сценария.

##### &nbsp; &nbsp; 11b Пользователь ввел слишком большое число
 &nbsp; &nbsp; &nbsp; &nbsp; 11b.1. пользователь вводит число >365;  
 &nbsp; &nbsp; &nbsp; &nbsp; 11b.2. система сообщает об ошибке;  
 &nbsp; &nbsp; &nbsp; &nbsp; 11b.3. происходит возврат на шаг 10 основного сценария.

#### Ограничения:
- бот может планировать задачи только на основе текущего часового пояса (UTC+3);

#### Исключения:
- соединение с БД потеряно  
  → система логирует ошибку и не сохраняет изменения;
- не удалось отправить сообщение пользователю  
  → система логирует ошибку

#### Минимальные гарантии:
- ошибки логируются для последующего анализа;
- в случае ошибки данные пользователя не перезаписываются некорректно

#### Гарантии успеха:
- в БД создана запись с актуальным chat_id и датой напоминания;
- напоминание успешно запланировано;
- пользователь получил подтверждение о выполненном действии.

## UC_02 Отменить напоминания

#### Цель
Отменить запланированные напоминания

#### Инициатор
Пользователь

#### Предусловие:
- в системе есть запланированные напоминания

#### Триггер
Пользователь отправляет команду `/cancel`

#### Основной сценарий:
1. система принимает команду;
2. система отменяет запланированные напоминания;
3. система отправляет подтверждение пользователю.

#### Исключения:
- ошибка доступа к базе данных  
  → система логирует ошибку и не изменяет данные пользователя;
- ошибка при работе планировщика (JobQueue)  
  → система логирует ошибку, часть задач может остаться активной;
- ошибка отправки сообщения пользователю  
  → система логирует ошибку, но операция отмены считается выполненной.

#### Ограничения:
- система использует встроенный планировщик задач;
- удаляются только задачи, привязанные к текущему chat_id.

#### Минимальные гарантии:
- данные пользователя не повреждаются;
- ошибки фиксируются в логах для последующего анализа.

#### Гарантии успеха:
- все активные напоминания пользователя удалены из планировщика;
- связанные данные пользователя удалены из базы данных;
- пользователь получил подтверждение выполнения операции.

## UC_03 Проверить статус напоминаний

#### Цель
Проверить, есть ли запланированные напоминания

#### Инициатор
Пользователь

#### Триггер
Пользователь отправляет команду `/status`

#### Основной сценарий:
1. система принимает команду;
2. система проверяет наличие запланированных напоминаний пользователя;
3. если напоминания существуют, система сообщает пользователю дату и время следующего напоминания и предлагает внести изменения (UC_05 "Изменить дату напоминания").

#### Альтернативный сценарий

##### &nbsp; &nbsp; 3а Запланированные напоминания отсутствуют
 &nbsp; &nbsp; &nbsp; &nbsp; 3а.1 система сообщает пользователю об отсутствии запланированных напоминаний и предлагает запланировать  
&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; (с кнопками согласия и отказа);  
 &nbsp; &nbsp; &nbsp; &nbsp; 3а.2.1 пользователь соглашается запланировать напоминание;  
 &nbsp; &nbsp; &nbsp; &nbsp; 3а.3.1 активируется UC_01 "Установить регулярные напоминания"

##### &nbsp; &nbsp; 3а.2а Пользователь отказывается
&nbsp; &nbsp; &nbsp; &nbsp; 3a.2а.1 пользователь отказывается от планирования;  
&nbsp; &nbsp; &nbsp; &nbsp; 3а.2а.2 система сообщает пользователю о действиях в случае, если он передумает.

#### Исключения:
- ошибка доступа к базе данных  
  → система логирует ошибку и не возвращает данные пользователю;
- ошибка обработки даты/времени  
  → система логирует ошибку и отправляет сообщение об ошибке;
- ошибка отправки сообщения  
  → система логирует ошибку, сценарий считается незавершённым с точки зрения пользователя.

#### Ограничения:
- отображается только ближайшее запланированное напоминание.

#### Минимальные гарантии:
- ошибки не приводят к созданию или удалению напоминаний;
- все ошибки логируются.

#### Гарантии успеха:
- пользователь получает актуальную информацию о статусе напоминаний;
- пользователь получает возможность изменить или создать напоминание.

#### Связанные UC:
- include: UC_01 "Установить регулярные напоминания"
- extend: UC_05 "Изменить дату напоминания"

## UC_04 Отложить напоминания

#### Цель
Отложить текущее напоминание на более позднее время

#### Инициатор
Пользователь

#### Триггер
Пользователь нажимает одну из кнопок отложенного напоминания

#### Предусловие:
- у пользователя есть активное напоминание;
- система отправила напоминание с кнопками:  
`«Напомнить через час»`, `«Напомнить завтра»`.

#### Основной сценарий:
1. система принимает нажатие кнопки пользователя;
2. система определяет выбранный вариант отложенного напоминания;
3. система удаляет старую задачу и создает новую с обновленным временем;
4. система отправляет пользователю подтверждение об успешном переносе.

#### Ограничения:
- доступны только фиксированные варианты переноса (1 день, 1 час).

#### Исключения:
- система не может обновить данные напоминания
  → логирует ошибку и уведомляет пользователя о временной недоступности функции.

#### Минимальные гарантии:
- при ошибке данные пользователя не искажаются;
- исходное напоминание сохраняется, если новое не удалось создать;
- все ошибки логируются.

#### Гарантии успеха:
- новое напоминание успешно запланировано;
- пользователь получил подтверждение переноса.

Пользователь

##### &nbsp; &nbsp; 3b Дата в прошлом

#### Ограничения:
